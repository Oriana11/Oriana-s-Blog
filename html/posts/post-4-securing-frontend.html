Securing Your Frontend — Headers, CSP, and Safe Coding Habits

By Oriana • Mar 20 2026 • 7 min read • #security #networking

Frontend security is underestimated. Most developers think of security as a backend concern — authentication, databases, server hardening. But your browser is a powerful execution environment that attackers target directly, and the decisions you make in HTML, CSS, and JavaScript have real security consequences. This post covers the practical steps to harden a frontend from the ground up.

Security Headers — Your First Line of Defence

HTTP response headers are configuration signals that tell the browser how to behave when rendering your page. They cost nothing to add and prevent entire categories of attack. These are the ones that matter most:

# Content-Security-Policy: restricts what scripts/styles can execute
Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' https://fonts.googleapis.com; font-src 'self' https://fonts.gstatic.com; img-src 'self' data:; object-src 'none'

# Prevents browsers from MIME-sniffing responses
X-Content-Type-Options: nosniff

# Blocks the page from being embedded in iframes (clickjacking)
X-Frame-Options: DENY

# Forces HTTPS for future visits (set after you're confident TLS is stable)
Strict-Transport-Security: max-age=63072000; includeSubDomains; preload

# Controls how much referrer info is shared when navigating away
Referrer-Policy: strict-origin-when-cross-origin

# Restricts browser API access (camera, microphone, geolocation, etc.)
Permissions-Policy: camera=(), microphone=(), geolocation=()

Content Security Policy In Depth

CSP is the single most powerful tool for preventing Cross-Site Scripting (XSS). It works by whitelisting the sources that are allowed to load scripts, styles, images, and other resources. Any resource from a non-whitelisted source is blocked by the browser before it executes.

The key principle: start with default-src 'self' and loosen only what you need. Avoid 'unsafe-inline' for scripts — it defeats most of CSP's value. If you use a framework that generates inline styles (like many CSS-in-JS solutions), consider a nonce-based or hash-based approach instead.

Sanitise Everything That Goes Into the DOM

Whenever you render user-supplied content — a comment, a username, a search result — you must treat it as untrusted. Never set innerHTML directly from external data. Use textContent for text, or a purpose-built sanitiser library like DOMPurify for HTML content that genuinely needs to render markup.

// BAD — opens XSS if content contains <script> or event handlers
element.innerHTML = userComment;

// GOOD — text only, no HTML interpretation
element.textContent = userComment;

// GOOD — HTML allowed, but sanitised first
import DOMPurify from 'dompurify';
element.innerHTML = DOMPurify.sanitize(userComment);

Safe Use of Third-Party Scripts

Every third-party script you load is code you haven't reviewed running in your page, with full access to the DOM and any data it contains. Minimise the number of third-party scripts. For scripts you do load, use Subresource Integrity (SRI) to ensure the file hasn't been tampered with:

<script
  src="https://cdn.example.com/library.js"
  integrity="sha384-abc123..."
  crossorigin="anonymous">
</script>

Cookies and LocalStorage

Session tokens and sensitive data should live in cookies with HttpOnly (prevents JavaScript access), Secure (HTTPS only), and SameSite=Strict (prevents CSRF). LocalStorage and sessionStorage are accessible to any JavaScript running on your page — including injected code — so they are not suitable for security-sensitive data like auth tokens.

Dependency Hygiene

Run npm audit regularly and do not dismiss high-severity findings. Outdated dependencies with known CVEs are a major vector for supply-chain attacks. Use tools like Dependabot or Renovate to auto-open PRs for dependency updates. Lock your dependency tree with a lockfile and commit it to version control.

The Mindset Shift

Frontend security is not a checklist you run once at launch. It is an ongoing posture. New browser APIs introduce new attack surfaces. New dependencies bring new risks. The best habit you can build is to ask "what could go wrong here?" every time you ship code that touches user data or renders external content. The headers, the sanitisation, the cookie flags — those are just the mechanisms. The mindset is the real security control.
