Networking Fundamentals Every Developer Should Know

By Oriana • Mar 20 2026 • 7 min read • #networking #cloud

You do not need to be a network engineer to write good software, but you do need a working mental model of how data moves from one machine to another. Misunderstanding the network is responsible for an enormous share of bugs, security vulnerabilities, and performance problems that plague production systems. This article covers the concepts that come up most often in day-to-day development work.

The OSI Model in Practice

The OSI model is seven layers of abstraction that describe how a packet of data travels from an application on one machine to an application on another. As a developer, you primarily operate at Layer 7 (Application — HTTP, WebSockets, DNS), Layer 4 (Transport — TCP, UDP), and occasionally Layer 3 (Network — IP addressing and routing). Understanding which layer a problem lives at saves significant debugging time.

When your HTTP request times out, is the problem a dead server (Layer 7), a broken TCP handshake (Layer 4), or an unreachable IP (Layer 3)? Each diagnosis requires different tools: curl for Layer 7, telnet or nc for Layer 4, ping and traceroute for Layer 3.

TCP vs UDP: Choosing Correctly

TCP guarantees delivery and ordering at the cost of overhead — the three-way handshake, acknowledgements, retransmission. It is the right choice for anything where losing data is unacceptable: HTTP, database connections, file transfers. UDP skips all of that and just fires packets. It is right for latency-sensitive streams where a dropped packet is preferable to a delayed one: video calls, online games, DNS lookups.

nc -zv example.com 443

DNS Matters More Than You Think

DNS converts human-readable hostnames into IP addresses. It is cached aggressively at every layer — OS, browser, resolver — which means a misconfigured TTL can haunt you for hours after a deployment fix. Know your TTLs: set them low (60–300 s) several days before a migration, make the change, then raise them again. And always query the authoritative nameserver directly when debugging, not your local cache.

HTTPS, TLS, and What a Certificate Actually Does

TLS (often still called SSL) provides two things: encryption of data in transit and authentication that you are talking to who you think you are. Authentication is done via certificates signed by a trusted Certificate Authority. When a browser says "Not Secure", it means one of those two guarantees is missing or untrusted. For development, use tools like mkcert to issue locally-trusted certificates rather than disabling TLS entirely — skipping TLS locally trains you towards bad habits and masks real issues.

Cloud Networking Basics

Cloud platforms like AWS, Azure, and GCP map closely onto the same concepts: Virtual Private Clouds (VPCs) are your Layer 3 subnets, Security Groups are stateful firewall rules at the host level, and load balancers operate at Layer 4 or Layer 7. If traffic is not reaching your service, work down from Layer 7 — is the app healthy? Is the security group allowing the port? Is the route table pointing at the right gateway? Systematic thinking by layer beats guessing every time.
